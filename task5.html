Task 05

Preemptive vs. Non-Preemptive Scheduling

There are two ways an OS can decide when to take the CPU away from a running process. This page explains the difference between them.

The core difference

The only question that separates these two is: can the OS forcefully take the CPU away from a running process?

If yes, it is preemptive. If no, it is non-preemptive.

Preemptive Scheduling

How it works: The OS acts as a strict manager. It can forcefully pause a running process, take the CPU away, and hand it to another process at any time.

Analogy: A moderated debate. The moderator gives each speaker a time limit. When the timer goes off or breaking news happens, the moderator cuts the microphone and gives it to the next person. The old speaker's spot is saved for later.

Real world benefit: This is why your mouse cursor still moves and your music still plays even when your browser is loading a heavy page. The OS rapidly preempts the browser to give the mouse and audio a fraction of a second to do their jobs.

Used by: Windows, Linux, macOS, iOS, Android (virtually all modern OSes).

Non-Preemptive (Cooperative)

How it works: Once a process gets the CPU, it holds onto it until it is completely finished, or until it voluntarily gives it up (like when it asks for I/O).

Analogy: A polite town hall meeting. Once someone gets the microphone, they cannot be interrupted. Even if there is an emergency, everyone must wait until the current speaker willingly steps down.

The problem: If a program gets stuck in an infinite loop or takes a very long time to calculate something, the entire computer freezes. Other programs simply cannot run.

Used by: Older systems, some simple embedded systems, cooperative multitasking (old Windows 3.x).
