Context Switching

How the operating system swaps one process out of the CPU and brings another one in, without losing any work.

What is Context Switching?

Context switching is when the CPU stops running one process and starts running another. To do this safely, the OS must save the current state of the running process, then load the state of the new process. This way, the old process can continue later from exactly where it stopped.

The "state" here means everything the CPU was working on: register values, program counter, flags, and other important info. All this data is stored inside the Process Control Block (PCB).

How it works (step by step)

  1. Process A is running on the CPU.
  2. An interrupt or system call happens (for example, the time slice ends).
  3. The OS saves Process A's state into its PCB (registers, program counter, flags).
  4. The OS loads Process B's state from its PCB into the CPU.
  5. Process B starts running from where it last stopped.

The Hidden Cost

Overhead: While the OS is doing a context switch, the CPU is not doing any useful work for any program. This wasted time is called "pure overhead" or dispatch latency.

This means that switching too often actually slows down the system. Modern operating systems try hard to make context switching as fast as possible, sometimes using special hardware like multiple register sets to speed it up.

When does it happen?

A context switch happens whenever:

  1. A higher priority process arrives (preemption).
  2. The running process's time slice ends.
  3. The running process makes a system call or asks for I/O.
  4. An interrupt happens (like a keyboard press or a disk finishing).
