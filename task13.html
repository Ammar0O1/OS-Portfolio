What is a Deadlock?

A deadlock is a situation in which two or more processes are unable to proceed because each is waiting for resources held by the other. It's a state of indefinite waiting where no process can proceed.

🔴 Critical Problem

In a deadlock state, no process makes progress. The system appears frozen, and manual intervention may be required.

Real-World Analogy

Two cars approach a single-lane bridge from opposite directions. Neither can proceed because the other car blocks the path, and neither is willing to back up. Both cars are stuck indefinitely.

In Computing

Process A holds Resource 1 and waits for Resource 2. Process B holds Resource 2 and waits for Resource 1. Neither can proceed—they are deadlocked.

Necessary Conditions for Deadlock

All four of these conditions must be present simultaneously for deadlock to occur:

# Condition Meaning
1 Mutual Exclusion Resources cannot be shared; only one process can use at a time
2 Hold and Wait Process holds resources while waiting for others
3 No Preemption Resources cannot be forcibly taken from a process
4 Circular Wait Circular chain of processes waiting for resources
💡 Key Insight

If any ONE of these conditions is eliminated, deadlock cannot occur. This principle guides all deadlock prevention strategies.

Classic Deadlock: Dining Philosophers

Five philosophers sit at a table with a bowl of rice and 5 chopsticks (one between each pair). Each philosopher needs 2 chopsticks to eat.

Deadlock Scenario:

  1. Each philosopher picks up left chopstick
  2. Each philosopher tries to pick up right chopstick
  3. No right chopsticks available (all held by left neighbor)
  4. All philosophers wait indefinitely ⚠️

Deadlock Prevention Strategies

Prevent deadlock by breaking one of the four necessary conditions:

🚫 Break Mutual Exclusion

Approach: Allow resource sharing
How: Use read-only copies, Spooling
Limitation: Not always possible (hardware limits)

🚫 Break Hold and Wait

Approach: Request all resources at once
How: Process requests all resources before proceeding
Limitation: Low resource utilization, starvation

🚫 Break No Preemption

Approach: Allow resource preemption
How: Take resources from waiting process if needed
Limitation: Difficult for non-preemptible resources

🚫 Break Circular Wait

Approach: Resource ordering
How: Processes request resources in ascending order
Limitation: Requires coordination

Prevention Example: Resource Ordering

Prevent the bank transfer deadlock by always acquiring locks in the same order:

// DEADLOCK: Lock in different order void transfer_a_to_b(Account a, Account b) { lock(a); lock(b); // transfer } void transfer_b_to_a(Account b, Account a) { lock(b); // Different order = Deadlock! lock(a); // transfer }
// SAFE: Always lock in ID order void transfer(Account a, Account b, int amount) { Account first = (a.id < b.id) ? a : b; Account second = (a.id < b.id) ? b : a; lock(first); // Always lock smaller ID first lock(second); // transfer amount unlock(second); unlock(first); }

Deadlock Detection and Recovery

Detection Methods

  • Resource Allocation Graph: Detect circular wait patterns
  • Banker's Algorithm: Check if state is safe before allocation
  • Timeout: Kill processes that exceed time limits
  • Wait-for Graph: Identify circular dependencies

Recovery Methods

  1. Process Termination: Kill one or more deadlocked processes
  2. Resource Preemption: Take resources from processes and reallocate
  3. Rollback: Restore processes to previous safe state
  4. Database Transactions: Automatic abort and retry

Summary: Deadlock Handling Strategies

Strategy Approach Pros Cons
Prevention Break a necessary condition Guarantees no deadlock Low resource utilization
Avoidance Banker's Algorithm (safe states) Better resource utilization Complex, overhead
Detection Detect then recover Flexible, no prevention overhead Requires recovery mechanism
Ignorance Assume deadlock won't happen Simplest, no overhead Risk of system hangs
