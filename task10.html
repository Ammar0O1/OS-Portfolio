Interprocess Communication: Message Passing Model

A simple visual explanation of how processes exchange information without sharing the same memory space.

Main idea

What is message passing?

Interprocess Communication (IPC) is how separate processes communicate with each other. In the message passing model, processes do not directly edit the same memory. Instead, one process sends a message and the operating system delivers it to another process.

In simple words, Process A gives a message to the OS, and the OS passes that message to Process B. The two processes do not touch each other's memory directly, which makes communication safer.

Basic message passing flow

1. SenderProcess A creates a message.
2. Send()A system call passes it to the OS.
3. Kernel queueThe OS stores or routes the message.
4. Receive()Process B asks for a message.
5. ReceiverProcess B reads and handles it.

Send operation

A process uses send(destination, message). The message is passed to the OS, not written into the receiver's private memory.

Receive operation

A process uses receive(source, message). If a message exists, it is delivered. If not, the process may wait.

Kernel support

The operating system checks permissions, keeps queues, wakes blocked processes, and prevents one process from corrupting another.

Direct vs. indirect communication

Direct message passing

The sender names the exact receiver, for example: send(ProcessB, "data").

  • Simple for one-to-one communication.
  • Sender and receiver must know each other.
  • Common in client-server request/reply designs.

Indirect message passing

Messages are sent to a mailbox or port, for example: send(mailbox1, "data").

  • Good when many processes may send or receive.
  • Processes communicate through a shared mailbox name.
  • The OS manages the mailbox queue.

Blocking and non-blocking behavior

Operation type Meaning Simple example
Blocking send The sender waits until the message is accepted or received. A process waits until the receiver takes the request.
Non-blocking send The sender sends the message and continues immediately. An app sends a print job and keeps working.
Blocking receive The receiver waits if no message has arrived. A server waits for the next client request.
Non-blocking receive The receiver checks for a message and continues if none exists. A program checks notifications without stopping.

Advantages

  • Works between processes on the same computer or across a network.
  • Safer than shared memory because processes remain separated.
  • Easier to understand because communication is explicit: send and receive.
  • Useful for client-server systems, microkernels, pipes, sockets, and distributed systems.

Limitations

  • Usually slower than shared memory because the OS must copy or transfer messages.
  • Queues can become full if messages are produced faster than they are received.
  • Programs must handle waiting, missing messages, and message ordering carefully.

Short real-life example

Imagine a web browser process asking a network service process to download a file. The browser sends a request message such as "download this URL". The network service receives it, does the work, and can later send a result back. The important point is that the request is sent as a message instead of one process directly changing another process's memory.

Key takeaway

Message passing is an IPC method where processes communicate by exchanging messages through the operating system. The OS manages the delivery, so each process can stay separate while still sharing information.
