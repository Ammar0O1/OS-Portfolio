dup() and dup2()

Two system calls that copy file descriptors. They are the trick behind how the shell redirects output, like when you type a command and send its output to a file.

First, what is a file descriptor?

When a program opens a file, the operating system gives it a small number to refer to that file. This number is called a file descriptor (fd). Instead of using the file name every time, the program just uses this number.

Every process starts with three file descriptors already open:

FD Name Used for
0 stdin Standard input (keyboard)
1 stdout Standard output (screen)
2 stderr Standard error (screen)

What does dup() do?

The dup() system call takes a file descriptor and makes a copy of it. Two important things to remember:

  1. Creates an alias: the new descriptor points to the exact same open file as the original.
  2. Auto-assigns the index: it automatically picks the lowest available (unused) number for the new copy.

In this example we make a twin of file descriptor 1 (the screen). Writing to either one does the same thing:

// FD 1 is always the screen (stdout)
// Make a twin of FD 1. It gets the next free number, like 3.
int screen_copy = dup(1);

// Now writing to '1' or 'screen_copy' does the same thing
write(1, "Hello from original!\n", 21);
write(screen_copy, "Hello from the twin!\n", 21);
close(screen_copy);

And here is the important detail: because both descriptors point to the same open file, the twin continues writing exactly where the original left off:

// fd1 points to "example.txt"
int fd1 = open("example.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);

// fd2 is a "twin" of fd1
int fd2 = dup(fd1);

write(fd1, "Hello ", 6);   // write with the original
write(fd2, "World!\n", 7); // twin continues where fd1 stopped

// example.txt now says: "Hello World!"
close(fd1);
close(fd2);

What does dup2() do?

The dup2() call lets you force a specific number for the new file descriptor (like forcing a file to take the slot for stdout). If that number is already open, dup2() automatically closes it for you first.

// Open your file
int file_fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);

// The magic line: force FD 1 (stdout) to point to file_fd
// It automatically closes the old FD 1 for you!
dup2(file_fd, 1);

// Now printf writes to output.txt instead of the screen
printf("Hello from printf using dup2!\n");
close(file_fd);

You can also redirect just the errors. Here stderr (FD 2) is sent to a log file, while normal output still goes to the screen:

// Open an error log file
int log_fd = open("errors.log", O_WRONLY | O_CREAT | O_TRUNC, 0644);

// Redirect stderr (FD 2) to the log file
dup2(log_fd, 2);

printf("This still shows up on your screen.\n");     // goes to screen (FD 1)
fprintf(stderr, "ERROR: Something went wrong!\n"); // goes to errors.log
close(log_fd);

This is exactly how output redirection works in the shell. When you type ls > out.txt, the shell uses dup2() to make file descriptor 1 (stdout) point to the file instead of the screen.

Why is this useful?

  1. Output redirection: send a program's output to a file instead of the screen (command > file).
  2. Input redirection: read a program's input from a file instead of the keyboard (command < file).
  3. Pipes: connect the output of one program to the input of another (cmd1 | cmd2).

The key idea: the program itself does not need to change. It still just writes to file descriptor 1. But because we redirected that descriptor, the output ends up somewhere else. This makes programs flexible and reusable.
